# ADR-0009 로그인 없는 개인화 전략

## 상태
확정

## 배경
즐겨찾기, 최근 사용 도구, 좋아요 기능을 구현하려면 사용자를 어떻게 식별할지 결정해야 한다.
이 서비스에는 회원가입·로그인 기능이 없다.

## 검토한 선택지

| 방식              | 설명                   | 문제점                             |
|-----------------|----------------------|---------------------------------|
| A. 회원가입 도입      | JWT 또는 세션 기반 로그인     | 구현 범위가 폭발적으로 늘어남. 포트폴리오 초점이 분산됨 |
| B. 익명 세션 ID     | 서버가 UUID를 발급, 쿠키로 유지 | 서버 세션 관리 필요. 브라우저 닫으면 소멸        |
| C. localStorage | 브라우저가 UUID 보관        | 서버 부담 없음. 브라우저 초기화 시 소멸 (허용 가능) |

## 결정
**C. localStorage 기반 클라이언트 개인화 + 글로벌 카운터(DB)**

- **즐겨찾기·최근 사용**: localStorage에 moduleId 배열 저장. 서버 API 없음.
- **좋아요 수**: ToolStats.like_count DB 컬럼에 글로벌 카운터로 저장.
  - 사용자가 좋아요를 눌렀는지 여부는 localStorage에 기록. 중복 방지는 클라이언트 책임.
- **사용 횟수**: ToolStats.use_count DB 컬럼. Job 생성 시 증가.

## 이유

이 서비스의 목적은 개발자 도구 허브다. 개인화는 "편의 기능"이지 핵심이 아니다.
브라우저 초기화 시 즐겨찾기가 사라지는 것은 이 서비스 수준에서 허용 가능한 트레이드오프다.

로그인 없이 이 정도 개인화를 구현한 설계 자체가 면접에서 설명할 수 있는 의사결정이다.

## 결과
- 서버: ToolStats 테이블(use_count, like_count) + Job 생성 시 use_count 증가 로직
- 클라이언트: localStorage key `devtoolbox_favorites`, `devtoolbox_recent`, `devtoolbox_liked`
- 개인 데이터가 서버에 저장되지 않으므로 개인정보 이슈 없음
