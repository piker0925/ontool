# ADR-0004 모듈 MVP 포함 범위

## 상태
업데이트 (2026-07-05: 모듈 수 30+로 확장, GIF 조건부 복귀)

## 배경
MVP에 포함할 모듈 수와 종류를 검토했다. 초기에는 무거운 파일 변환 모듈 4개로 시작했으나,
프로젝트 방향이 "개발자 일상 도구 허브"로 확장되면서 카테고리와 모듈 수가 대폭 늘었다.

## 결정
**30+ 모듈, 10개 카테고리 MVP 포함.**

무거운 모듈(Heavy — DB 큐, 비동기 워커, SSE)과 가벼운 모듈(Light — 동기 처리)로 분리.

| 구분 | 카테고리 |
|------|----------|
| Heavy | PDF, 이미지, 코드 생성, 아카이브 |
| Light | 포맷터, 텍스트, 보안/암호화, 개발 유틸, QR/바코드, 네트워크/API |

## 이유

이 프로젝트의 백엔드 복잡도는 모듈 코드가 아니라 **공유 인프라**에 있다.

| 영역 | 복잡도 위치 |
|------|------|
| 모듈 코드 | 낮음 — 라이브러리 호출 20~50줄 수준 |
| 큐/워커 구조 | 높음 — 공유 인프라 |
| SSE 상태 알림 | 높음 — 공유 인프라 |
| 배치 집계 | 높음 — 공유 인프라 |

모듈이 4개든 30개든 공유 인프라 코드는 동일하다. `@Component` 추가 하나로 새 모듈이 자동 등록되므로
모듈 수가 늘어도 추가 구현 부담이 거의 없다.

무거운 모듈이 큐·비동기·SSE 아키텍처를 실증하고, 가벼운 모듈이 개발자 일상 도구 허브로서의
실용성을 보여준다.

**WebP 제거 이유:** webp-imageio가 JNI 네이티브 바인딩(libwebp)을 사용한다.
Oracle Cloud 배포 환경(ARM64 Linux)에서 `UnsatisfiedLinkError` 발생 가능.

**GIF 생성 조건부:** 구현 시점에 JDK 25 호환 라이브러리(AnimatedGifEncoder 등) 확인 후 포함 여부 결정.
호환 라이브러리가 없으면 제외.

## 결과
- 모듈 추가: `@Component` 추가만으로 자동 등록. 별도 등록 코드 불필요
- Heavy/Light 구분은 `ToolModule.isHeavy()` 반환값으로 결정
- GIF, WebP는 확장 단계에서 호환성 확인 후 추가 가능
