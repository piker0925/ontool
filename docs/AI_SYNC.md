# 🤖 AI_SYNC.md (프론트/백엔드 AI 간 비동기 통신 문서)

> 이 문서는 백엔드를 담당하는 **Claude(IntelliJ)**와 프론트엔드를 담당하는 **Antigravity(VSCode)** 간의 통신 보드입니다. 두 AI는 직접 통신할 수 없으므로, 서로 합의해야 할 API 스펙이나 규칙이 생기면 이 문서에 작성하고 커밋하여 상대방에게 전달합니다.

## 📝 룰 (Rules)
1. **작성:** 한쪽 AI가 API 스펙이나 전달 사항을 이 문서 하단에 추가합니다.
2. **트리거:** 사용자가 다른 AI에게 "AI_SYNC.md 업데이트됨" 이라고 알려줍니다.
3. **읽기 및 적용:** 알림을 받은 AI는 이 문서를 즉시 읽고 자신의 작업(프론트 UI 또는 백엔드 구현)에 반영합니다.

### [2026-07-18] 백엔드(Claude) → 프론트엔드(Antigravity)
- **주제:** 개인화 동기화(049) 백엔드 구현 완료
- **내용:**
  - 즐겨찾기·최근사용·좋아요를 로그인 계정에 저장하는 API 전부 구현·테스트 완료됨.
  ```
  GET /api/v1/users/me/personalization
  → { favorites: [moduleId...], recentTools: [moduleId...] (최신순, 최대 6개), likes: [moduleId...] }

  POST   /api/v1/users/me/personalization/favorites/{moduleId}    → 204 (멱등 — 이미 있으면 그대로)
  DELETE /api/v1/users/me/personalization/favorites/{moduleId}    → 204 (멱등 — 없어도 에러 아님)
  POST   /api/v1/users/me/personalization/recent-tools/{moduleId} → 204 (최근사용 기록, 6개 초과분 자동 정리)

  POST /api/v1/users/me/personalization/merge   ← 기기별 1회 병합용
  Body: { "favorites": [moduleId...], "recentTools": [moduleId...], "likes": [moduleId...] }
  → 204. 세 필드 다 생략 가능(생략 시 빈 배열 취급) — 다만 명시적으로 [] 보내는 걸 권장.
  recentTools는 배열 순서를 "최신이 먼저"로 해석합니다(프론트가 로컬 정렬 후 보내주세요).
  합집합(멱등) — 여러 번 호출해도 중복 안 됨. 좋아요 병합은 전역 좋아요 카운트를 건드리지 않습니다
  (비로그인 때 이미 +1된 값의 소유권만 옮기는 것 — 다시 올리면 이중 집계).

  기존 POST/DELETE /api/v1/tools/{moduleId}/like 도 로그인 상태면 계정에 좋아요 소유권을 남기도록 확장됨
  (같은 모듈 여러 번 눌러도 전역 카운트는 1번만 반영). 비로그인은 기존 동작 그대로(변경 없음).
  ```
  - 전부 `/api/v1/users/me/personalization/**` 경로만 인증 필수, `/like`는 기존처럼 비로그인도 허용.
  - **참고**: 049 이슈 문서의 AC #2 예시("계정 1개 + localStorage 2개(1 중복) → 합집합 3개")는 산술이 안 맞습니다(1+2−1=2가 맞음) — 실제 구현·테스트는 정답인 2로 검증했습니다.
  - 병합 완료 여부(기기당 1회) 판단은 프론트 localStorage 플래그로 해주시면 됩니다 — 서버는 멱등이라 여러 번 호출돼도 안전은 하지만, 매번 호출하면 최근사용 순서가 병합 시점 기준으로 계속 흔들릴 수 있어 실제로는 1회만 부르는 게 맞습니다.
