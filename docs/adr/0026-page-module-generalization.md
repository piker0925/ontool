# ADR-0026 페이지 카탈로그 통합 — 명부·틀·알맹이 3층 분리

## 상태

확정 (2026-07-17, 같은 날 비판 검토로 1차 안 개정) — 구현 전. 1차 IA 공사(이슈 044)에서 메타 필드만 도입하고, 셸(틀) 구현은 각 기능 이슈에서.

> **개정 이력**: 1차 안은 "페이지 모듈 일반화 — kind가 렌더 계약을 결정"이었다. 비판 검토에서 ① kind 중앙 스위치는 새 형태마다 기존 코드 수정을 강제하고(개방-폐쇄 위반), ② kind 경계가 흐려(오디오는 studio인가, 실시간 미리보기 있는 tool인가) 분류 논쟁이 구현을 막으며, ③ "추상화 일반화"라는 서사가 실제로 공유되는 것(데이터)보다 과대하다는 문제가 확인돼 아래로 개정했다.

## 배경

v1의 "모듈화 컨셉"(입력·출력 동일시, 추가·삭제 간편)은 실제로는 두 계층의 결합이다:

- **명부(카탈로그)**: `toolConfigs` 메타 — id·이름·설명·키워드·카테고리(·v2 구역). 검색(⌘K)·즐겨찾기·최근 사용·구역 노출·stats·댓글은 전부 이 데이터만 읽는다.
- **틀(렌더)**: `/tools/:id` → 단일 `ToolPage.vue`가 "입력 → 실행 → 결과" 화면을 그린다.

v1에서는 모든 페이지가 도구였으므로 틀이 하나뿐이었고 두 계층의 결합이 문제가 안 됐다. v2에는 이 틀에 맞지 않는 페이지(미니게임·뽀모도로=상주형, 문서 뷰어=출력 없음, 오디오=실시간 조작)가 들어온다.

**중요한 사실 인정**: 게임과 도구가 공유하는 것은 **행위가 아니라 데이터(명부)뿐**이다. 공통 실행 인터페이스 같은 행위 공유를 강제하면 그게 바로 과추상화다. 따라서 이 결정은 "추상화를 한 단계 올리는 것"이 아니라 **명부를 통합 유지하면서 틀을 분리하는 것**이다.

## 결정

**3층 구조로 분리한다. 명부는 하나, 틀은 유형별로, 알맹이의 기존 모듈화는 그대로.**

```
층1  명부(카탈로그)   모든 페이지 공통 메타. 검색·즐겨찾기·구역·stats·댓글의 유일한 데이터원
층2  셸(틀)           ToolPage(입력→실행→결과) · GamePage(보드+시작/재시작) ·
                      ViewerPage(파일→렌더) · StudioPage(로드→실시간 조작→내보내기) — 틀당 1개
층3  콘텐츠(알맹이)   백엔드 ToolModule(Heavy)·프론트 유틸(Light)·개별 게임 로직·개별 뷰어 파서
                      ← v1 모듈화가 그대로 사는 곳. 백엔드 무변경
```

**렌더 결정은 `component` 참조로** (kind 스위치가 아니라):
- 명부의 각 항목이 자기 페이지 컴포넌트를 직접 가리킨다: `component: () => import('.../Game2048Page.vue')`. **생략 시 기본값 ToolPage** — 기존 도구 전부 무변경.
- 중앙 `switch(kind)` 없음 → 새 형태 추가가 기존 코드를 건드리지 않고, 분류 논쟁(오디오=tool? studio?)이 구현을 막지 않는다. 지연 로딩(게임 코드는 진입 시에만 다운로드)도 자연스럽다.

**`kind`는 분류 라벨로만** (`'tool' | 'game' | 'viewer' | 'studio'`, 기본 `'tool'`):
- 용도: 구역 홈 섹션 묶기("게임" 섹션), 뱃지 표시, 크롬 규칙(댓글·좋아요·사용수 UI를 붙일지). **렌더 결정권 없음** — 결정권 없는 라벨은 경계가 애매해도 사고가 안 난다.

## 트레이드오프 (의도적으로 받아들인 것)

| 얻는 것 | 내주는 것 | 완화 |
|---------|-----------|------|
| 새 형태 추가 시 기존 코드 무수정 | 틀 사용의 일관성이 강제가 아닌 컨벤션이 됨 (게임이 제멋대로 다른 틀을 쓸 수 있음) | 컨벤션 명문화: "game 콘텐츠는 GamePage 셸 안에, viewer는 ViewerPage 안에" — 코드 리뷰에서 지킨다 |
| 분류 논쟁이 구현과 분리됨 | "어떤 항목이 어떤 틀"의 전체 지도가 한 파일에 안 보임 | 명부가 곧 지도(component 필드) — 정렬·검토는 명부 파일에서 |
| 지연 로딩 자연스러움 | 명부가 순수 데이터가 아니게 됨(코드 참조 포함) — 서버 배포형 카탈로그로 확장 불가 | 우리 명부는 프론트 빌드에 포함된 정적 파일 — 해당 없음. 필요해지면 그때 분리 |

## 명시적 한계 (과대 해석 금지)

- **이것은 다형성이 아니다.** kind/component는 태그 달린 라우팅이다. 행위 다형성은 백엔드 `ToolModule`(Worker가 구현을 모른 채 `process()` 호출)에만 존재하며, 프론트에서 그것을 흉내 내지 않는 것이 이 결정의 핵심이다.
- **use_count는 kind 간 비교 불가.** "사용 1회"의 의미가 다르다(tool=실행, game=시작, viewer=파일 열기, studio=내보내기). 페이지별 카운트일 뿐 통합 랭킹 지표가 아니다 — 인기 목록을 만들 땐 kind별로 나누거나 tool만 쓴다.
- 면접 서사도 이대로: "행위는 일부러 공유하지 않았다 — 공유 실체가 카탈로그뿐임을 확인했고, 억지 공통 인터페이스가 더 나쁜 선택이라 데이터 계층만 통합했다."

## 가드레일

- kind는 4종 고정. 다섯 번째가 필요해 보이면 기존에 욱여넣지 말고 ADR로 결정.
- 셸은 유형당 1개. 개별 게임·뷰어가 자체 셸을 만들면 리뷰에서 반려 — "메타 한 줄 + 콘텐츠 컴포넌트 하나 = 새 항목"의 비용 균일성이 이 컨벤션에 달려 있다.
- 1차(044)에서는 `kind`(전부 `'tool'`)와 `component`(전부 생략=ToolPage) 필드만 도입해 회귀 없이 스키마를 먼저 깐다.
