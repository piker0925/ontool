# ADR-0014 파일 수명·청소 정책

## 상태
확정

## 배경

Heavy 모듈은 업로드된 입력 파일을 `uploads/temp/{tempId}/`에, 처리 결과를 `uploads/{jobId}/result.*`에 저장한다. ADR-0011은 운영에서 Cloudinary 사용을 전제했으나, 실제 구현은 `LocalFileStorage`가 `@Profile({"local","prod"})`로 **운영에서도 로컬 디스크**(Docker `uploads_data` 볼륨)를 쓴다. 따라서 디스크가 무한히 차는 것을 막을 청소 정책이 실질적으로 필요하다.

기존 청소(`TtlCleanupScheduler`)에는 세 가지 구멍이 있었다:

1. **입력 임시파일이 결과와 같은 24h TTL까지 방치됐다.** 입력은 워커 처리 순간까지만 필요한데 결과와 함께 만료 청소를 기다렸다.
2. **Job row가 참조하는 파일만 지웠다.** 업로드 성공 후 Job 생성 실패로 생긴 고아 파일, 그리고 로컬 `create-drop`으로 job 테이블이 초기화되면 참조를 잃은 파일이 **영구히 남았다.**
3. **삭제 실패를 조용히 무시(`catch (IOException ignored)`)**했고, 보존시간 24h가 상수로 코드에 박혀 로컬에서 청소 동작을 검증할 수 없었다.

## 검토한 선택지

| 방식                          | 설명                           | 문제점                        |
|-----------------------------|------------------------------|----------------------------|
| A. 현행 유지 (Job row 기준 만료 삭제) | 만료 Job의 파일만 삭제               | 고아 파일 방치, 입력 24h 낭비, 검증 불가 |
| B. 외부 배치 크론으로 디렉토리 정리       | OS cron이 오래된 파일 삭제           | 앱 밖 인프라 의존, 배포마다 별도 설정     |
| C. 앱 내 수명 분리 + mtime 안전망    | 입력 즉시삭제 + 결과 TTL 설정화 + 고아 스윕 | 스윕 구현 필요하지만 앱에 자족적         |

## 결정

**C. 파일 종류별 수명 분리 + mtime 기반 폴백 스윕**

1. **입력 임시파일 — 처리 직후 즉시 삭제.** `JobWorker`가 DONE/FAILED 후 `finally`에서 입력 `temp/{tempId}` 디렉토리를 삭제한다(상태 저장을 먼저 하고 삭제).
2. **결과 파일 — 설정 TTL.** 보존시간을 `storage.result-ttl`로 분리한다. **기본값을 코드에 두지 않고 yaml 단일 진실원**으로 둔다(운영 `PT1H`, 로컬 `PT2M`). 값이 없으면 앱이 시작 시 실패(fail-fast). `JobService.create`가 `expiresAt = now + result-ttl`을 지정한다.
3. **고아 파일 — mtime 안전망 스윕.** `OrphanFileSweeper`가 Job row 존재 여부와 무관하게 최종수정시각이 TTL보다 오래된 파일을 삭제하고 빈 디렉토리를 정리한다. `TtlCleanupScheduler`가 Job 기준 삭제에 더해 이 스윕을 호출한다.
4. **삭제 실패는 `log.warn`으로 관측**한다(조용한 무시 제거). 개별 삭제가 실패해도 다음 스윕이 mtime으로 재수거한다.

## 이유

- **입력 즉시삭제**: 입력은 처리 후 쓸모가 없다. 24h 방치는 순수 낭비였다.
- **TTL 설정화 + fail-fast**: 운영/로컬이 서로 다른 보존시간을 갖고, 상수 대신 yaml 한 곳에서 관리한다. 도달불가한 기본 상수(구 24h)를 제거해 "실제 값이 어디 있는지" 혼동을 없앤다.
- **mtime 스윕**: JPA 엔티티(`Job`)는 스프링 설정을 못 읽으므로 만료 계산은 서비스가 담당하고, Job row가 사라져도 파일은 mtime으로 청소되어 로컬 `create-drop`·업로드 실패 시나리오를 모두 흡수한다. 외부 크론(안 B) 없이 앱이 스스로 정리한다.

## 결과

- 정상 운영에서 디스크 사용량은 "최근 TTL(운영 1h) 트래픽" 수준에서 평형을 이룬다. 별도 배치 크론 불필요.
- `Job.onCreate`는 `createdAt`만 세팅한다. `JobService`를 거치지 않고 저장되는 Job(테스트)은 `expiresAt`을 직접 지정해야 한다.
- ADR-0011의 "Job 만료 시 `FileStorage.delete()` 호출로 파일 정리"를 본 ADR이 대체·확장한다.

## 추록 (2026-07-18, 050 작업 이력)

**결정 변경: 회원 Job의 row는 TTL 만료 후에도 영구 보존한다.** 파일 삭제 정책(위 1~4)은 회원·익명 무관 동일하게 적용되지만, **row 삭제는 익명 Job(`user_id IS NULL`)에만** 적용한다. 회원 Job(`user_id`가 있는 Job, ADR-0024)의 row는 작업 이력(`GET /api/v1/users/me/jobs`) 조회를 위해 삭제하지 않는다.

- **배경**: 050 구현 중 이슈 명세가 "Job row는 만료돼도 전부 영구 보존된다"고 전제했으나, 실제 `TtlCleanupScheduler`와 본 ADR 원안 모두 만료 Job의 row를 무조건 삭제하고 있어 명세와 코드가 모순됐다. 사용자에게 확인한 결과 "로그인 유저 Job은 row 영구보존"으로 확정(익명 Job은 기존대로 삭제).
- **파일은 여전히 삭제된다**: 회원 Job이라도 `resultKey`가 가리키는 실제 파일은 TTL이 지나면 스케줄러가 삭제한다. row만 남기고 파일은 지운다 — 따라서 `resultKey != null`이라고 파일이 실재한다고 가정하면 안 된다. `JobService.isExpired(job)`으로 만료 여부를 먼저 확인해야 한다(`GET /api/v1/jobs/{id}/result`, `GET /api/v1/users/me/jobs` 둘 다 이 가드를 쓴다). `resultText`는 파일이 아니라 DB 컬럼이라 만료와 무관하게 유지된다.
- **결과**: 회원 Job 테이블은 시간이 지나며 무한히 커진다(설계상 트레이드오프 — 작업 이력 기능의 요구사항). 대량 삭제·아카이빙 정책은 아직 없음 — 필요해지면 별도 ADR로 다룬다. 055-②(회원 탈퇴)에서는 Job row를 삭제하지 않고 `user_id`를 null로 익명화하는 방식을 검토 중이다.
