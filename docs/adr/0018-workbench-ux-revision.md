# ADR-0018 Workbench UX 개정 — 사이드바 상시 표시·테마 드롭다운·색상 계열 마무리

## 상태
확정 (2026-07-14)

## 배경

ADR-0015로 Workbench 리디자인을 마친 뒤 실사용 중 세 가지 문제가 추가로 드러났다.

1. **사이드바가 예고 없이 사라짐**: ADR-0015 §4에서 도입한 `lg`(1024px) 미만 오버레이 드로어가, 모니터 해상도와 무관하게 브라우저 창 크기·줌 레벨·개발자 도구 도킹만으로도 뷰포트 CSS 폭이 1024px 밑으로 내려가면 사이드바를 숨겼다. 이 프로젝트는 모바일 대응이 공식 요구사항이 아니므로(ADR-0015 §4, CONTEXT.md) 애초에 드로어라는 장치 자체가 실이득 없이 "가끔 사라지는 버그"로만 체감됐다.
2. **테마 토글의 3번째 옵션(시스템)이 발견 불가**: 아이콘 하나로 라이트→다크→시스템을 순환하는 버튼은 상태를 호버 툴팁으로만 노출해, "시스템" 옵션의 존재 자체를 알 방법이 없었다.
3. **사이드바-본문 대비 부족 + "포인트 색이 없다"는 지적**: ADR-0015 §5(카드 아이콘 무채색 통일)의 취지는 유지하되, 사이드바와 본문의 시각적 구분과 인터랙션 피드백(hover/active)이 부족하다는 반복 피드백이 있었다.

## 결정

### 1. 사이드바 상시 표시 (ADR-0015 §4 대체)

`drawerOpen` 상태·백드롭·모바일 톱바(햄버거·검색 버튼)·관련 워처를 전부 제거하고, `<aside>`를 뷰포트 폭과 무관한 정적 flex 자식으로 변경했다. 브레이크포인트를 낮추는 안(예: `md`)도 검토했으나, 모바일 대응이 공식 요구사항이 아닌 이상 "얼마나 좁아야 숨길지"를 튜닝하는 것보다 애초에 숨기지 않는 편이 버그 표면을 줄인다고 판단했다.

### 2. 테마 컨트롤 — 순환 버튼 → 드롭다운 메뉴 (ADR-0015 §4 서술 대체)

3단(라이트/다크/시스템) 전환에서 일반적으로 쓰이는 패턴인 드롭다운 메뉴로 교체했다. shadcn-vue `dropdown-menu`를 신규 설치(`npx shadcn-vue add dropdown-menu`)하고 `DropdownMenuRadioGroup`/`DropdownMenuRadioItem`으로 세 옵션을 라벨과 함께 나열, 현재 선택엔 체크 아이콘을 표시한다.

**시도했다가 같은 날 되돌린 것**: 테마 전환 시 클릭 지점에서 원형으로 퍼지는 리빌 애니메이션(View Transitions API, `document.startViewTransition` + `clip-path: circle()`)을 한때 추가했으나, 실사용해보니 "너무 화려해서 정신없다"는 피드백을 받았다. YouTube·Naver·GitHub 등 대형 서비스가 테마 전환에 애니메이션을 쓰지 않거나 CSS 변수를 즉시 교체하는 것이 일반적이라는 점을 근거로, 속도를 늦추는 절충 대신 완전히 제거했다 — 원형으로 퍼지는 공간적 모션 자체가 시선을 끄는 요소라 느리게 해도 "정신없음"은 해소되지 않는다고 판단했기 때문이다. **다시 제안되더라도 이 이유로 한 번 기각됐다는 점을 참고할 것.**

### 3. 색상 전략 마무리 (ADR-0015 §5 연장, 카테고리 색상 재도입은 재기각)

"포인트 색이 하나도 없다"는 지적으로 아래 옵션을 비교했다: ①활성 카테고리에 코발트(`primary`) 액센트, ②Heavy/Light 배지 색 구분, ③카드 hover 반응 강화, ④로고 강조, ⑤아이콘 chip 무채색 대비, ⑥카테고리 3그룹 색 배정, ⑦카테고리 8색 전부. 이 중 ①③만 채택했다.

⑥⑦(카테고리별 색상)을 검토하던 중 `docs/adr/0015-workbench-redesign.md`에서 이미 한 번 시도했다가 뺀 이력을 재확인했다 — 원래 카드 아이콘이 카테고리별로 색이 있었는데 "무작위 분포해 정보를 전달하지 못한다"는 피드백으로 무채색 통일 + 섹션 헤더 구분으로 바뀐 것(ADR-0015 §5). **동일한 이유로 재기각**하며, `categoryConfig.ts`에 남아있는 8색 설정(`bg-indigo-50` 등)은 `.icon`만 쓰이는 죽은 코드로 그대로 둔다(카드가 `config.color`/`config.bg`를 계산은 하되 렌더링하지 않음 — 다음에 손댈 사람이 "왜 안 쓰지?"라고 묻지 않도록 이 ADR에 남긴다).

추가로 사이드바와 본문의 색상 계열(hue) 자체를 분리했다 — 기존엔 명도차만 0.017~0.02로 미미해 나란히 봐도 구분이 안 됐다:
- **라이트**: 본문은 웜 페이퍼(hue 85) 유지, 사이드바는 hue 260(쿨 그레이)으로 분리 + 명도도 0.955→0.9로 낮춤.
- **다크**: 본문은 그래파이트(hue 262, L 0.185) 유지, 사이드바는 같은 계열 안에서 더 어둡게(L 0.165→0.11) — VS Code 활동 표시줄처럼 "프레임은 어둡게, 작업 영역은 상대적으로 밝게"의 관례를 따름.

사이드바 active 상태는 무채색 `bg-sidebar-accent`에서 `bg-primary/10 text-primary`로, 카드 hover는 보더·그림자·이동량을 상향(`primary/35`→`primary/50`, `shadow`→`shadow-md` 등)해 hover(미리보기)와 active(선택됨)를 구분했다. 모든 대비 조정은 WCAG AA(4.5:1 이상) 기준으로 브라우저 canvas API 실측 후 확정했다.

## 이유

- 드로어·순환 버튼 모두 "기능은 있지만 발견 불가/예측 불가"였다는 공통점이 있다 — 이번 개정의 공통 원칙은 상태를 숨기지 않고 항상 보이게 하는 것.
- 애니메이션은 "디자인을 우선시"하라는 요청으로 시도했지만, 실사용 검증(코드 리딩이 아니라 브라우저에서 실제로 클릭)에서 문제가 드러났다 — 화려함과 사용성이 항상 비례하지 않는다는 근거로 남긴다.
- 카테고리 색상 재도입은 이미 한 번 실패한 접근을 반복하지 않기 위해 ADR-0015의 근거를 재확인하고 동일 결론을 냈다.
