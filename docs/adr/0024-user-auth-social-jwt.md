# ADR-0024 회원 인증 도입 — 익명 기본 + 소셜 로그인 + JWT

## 상태

확정 (2026-07-17) — 구현 전 (v2 2차). ADR-0009(로그인 없는 개인화)를 **부분 대체**한다: 익명 개인화(localStorage + 글로벌 카운터)는 그대로 유지하고, 그 위에 회원 계층을 추가한다.

## 배경

v2 포털 전환(ADR-0023)과 함께 로그인 기능 도입이 결정됐다. 제약 조건:

- 프론트(`*.vercel.app`)와 백엔드(`*.sslip.io`)가 **서로 다른 루트 도메인** — 크로스 사이트 쿠키(SameSite=None)는 브라우저의 서드파티 쿠키 차단 흐름 때문에 갈수록 취약하다. 커스텀 도메인 구매는 보류(브랜드명 미확정).
- 기존 시스템은 완전 익명: X-Client-Id 헤더(쿼터·공정 스케줄링), localStorage 개인화(즐겨찾기·최근·좋아요), 익명 댓글.
- 도구 사이트 특성상 "검색 → 바로 사용" 마찰 최소화가 중요.

## 결정

**익명 기본 + 로그인 혜택.** 모든 도구는 로그인 없이 지금처럼 사용 가능하다. 로그인은 강제가 아니라 부가 가치다.

**혜택 (v2 범위):**
1. **개인화 동기화** — 즐겨찾기·최근 사용·좋아요를 계정에 저장, 기기/브라우저 간 유지. 비로그인은 기존 localStorage 그대로.
2. **작업 이력** — 본인이 돌린 Heavy 작업(Job) 목록을 계정에서 조회. Job row 보존 정책은 **현행 그대로**(ADR-0014: 만료 시 파일만 삭제, row는 익명·회원 무관 전부 보존) — 정리 정책 변경이 아니라 회원에게 자기 row의 **조회권**을 부여하는 것.
3. **닉네임 댓글** — 회원 댓글에 닉네임 표시. 익명 댓글도 계속 허용.

**제외 (v2에서 안 함):** 쿼터/한도 차등 — AdmissionControl·레인 스케줄러·Rate Limiting(ADR-0019/0021) 세 곳에 회원 분기가 필요해 복잡도 대비 체감 가치가 없다. 실제 어뷰즈 관측 시 재검토.

**인증 수단: 소셜 로그인만 — 구글 + 카카오** (Spring Security OAuth2 Client). 비밀번호를 저장하지 않으므로 이메일 인증·비번 재설정·유출 대응 플로우가 통째로 생략된다.

**세션: JWT access + refresh** (도메인 미구매 상태에서 크로스 도메인 인증의 유일한 안정 경로):
- access 30분 / refresh 14일, `Authorization: Bearer` 헤더.
- refresh는 **회전(rotation)** — 재발급 시 새 refresh 발급·기존 폐기. 서버 DB에 저장해 강제 폐기(로그아웃·탈취 대응) 가능.
- 프론트 저장: localStorage. XSS 리스크는 수용 — 보안 스테이크가 낮은 도구 사이트이고, 짧은 access 수명 + refresh 회전으로 완화. (도메인 구매 시 httpOnly 쿠키 전환 재검토)
- OAuth2 로그인 성공 → 백엔드가 JWT 쌍 발급 → **프론트 콜백 라우트(`/auth/callback`)로 URL fragment(`#access=…&refresh=…`)에 담아 리다이렉트** — fragment는 서버 로그·Referer에 남지 않는다. 프론트 origin은 백엔드 설정 **`FRONTEND_URL`**(로컬 `http://localhost:5173` / 운영 Vercel 도메인, GitHub Secret 추가)로 관리.
- **refresh 동시성(멀티탭)**: 회전+재사용 거부를 문자 그대로 구현하면 탭 2개가 동시에 재발급할 때 정상 사용자가 강제 로그아웃된다. 방침 — ① 프론트: 재발급 요청 단일화(모듈 레벨 in-flight Promise 공유), ② 서버: 직전 refresh에 30초 유예(grace period)를 둬 경합 잔여 요청 흡수. 유예 초과 재사용은 탈취로 간주해 해당 사용자의 토큰 전체 폐기.
- **로그인 성공 후 HTTP 세션 무효화(2026-07-18 확정 — 해커 관점 재점검에서 발견)**: "쿠키 대신 JWT"라고 결정했지만 `SecurityConfig`가 `SessionCreationPolicy.STATELESS`를 지정한 적은 없다 — Spring Security의 `oauth2Login()`이 인가 요청(state)을 HTTP 세션에 보관하는 기본 동작을 그대로 쓰기 때문이다(`curl`로 `/oauth2/authorization/google` 호출 시 `JSESSIONID` 쿠키 발급을 실증 확인). 로그인이 성공하면 그 세션에 인증된 `SecurityContext`가 그대로 남는데, 로그아웃/토큰 폐기 로직은 JWT·refresh token만 다루고 이 세션은 건드리지 않아 — 로그아웃해도 안 죽는 별도 인증 채널이 될 수 있었다. `OAuth2LoginSuccessHandler`에서 JWT 발급 직후(세션의 유일한 용도가 끝나는 시점) 세션을 명시적으로 invalidate하는 것으로 막았다 — 실브라우저로 로그인 후 그 세션 쿠키만으로 `/api/v1/users/me` 직접 접근 시 401을 반환함을 확인. 더 근본적인 해법(STATELESS + 쿠키 기반 authorization-request-repository로 교체)은 이번엔 범위 밖으로 뺐다 — 지금 방식이 부작용 없이 문제를 닫기 때문.

**닉네임:** 소셜 프로필명을 기본값으로 자동 설정, 변경 가능, 중복 허용(커뮤니티 기능이 댓글뿐이라 유일성 불필요). **제약**: 트림 후 2~20자·빈 문자열 불가, DB `VARCHAR(20)`(utf8mb4). 소셜 프로필명이 20자 초과면 잘라서 기본값으로. 금칙어 필터는 보류(admin 댓글 삭제로 대응).

**익명→회원 전환(병합 시맨틱 — 기기별 1회):** 어떤 브라우저에서든 "그 브라우저에서 이 계정으로 처음 로그인"할 때, 그 브라우저 localStorage의 즐겨찾기·최근·좋아요를 서버로 **합집합 병합 업로드**한다(계정 기준 1회가 아님 — 두 번째 기기의 익명 데이터도 그 기기 첫 로그인 때 병합). 병합 완료 여부는 브라우저별 localStorage 플래그(`merged:{userId}`)로 기록. 병합 시 전역 좋아요 카운터는 재호출하지 않는다(이미 +1된 기록의 소유권 이전일 뿐). localStorage 원본은 지우지 않는다(로그아웃 후 익명 모드 복귀용). 익명 시절의 **Job 이력은 연결하지 않는다**(X-Client-Id↔계정 매핑은 하지 않음 — 단순화).

**계정 연동은 하지 않는다(2026-07-18 확정 — 마이페이지/관리자 그릴링 세션):** 같은 사람이 구글로도 카카오로도 로그인하면 `(provider, provider_id)` 유일성 때문에 **완전히 별개인 User row 2개**가 생기고, 이 둘을 나중에도 합치지 않는다. 계정 연동(하나의 사람=하나의 계정)을 만들려면 병합 UX·병합 시 개인화 데이터 충돌 해소 정책이 새로 필요한데, v2 범위에서는 복잡도 대비 이득이 낮다고 판단해 명시적으로 범위에서 뺐다. 되돌리기 어려운 결정이라(사용자들이 이미 두 계정을 따로 써온 뒤에 합치려면 마이그레이션이 필요) 여기 기록해둔다.

**관리자:** 기존 HTTP Basic(ADR-0010) 유지 — 회원 체계와 분리. 관리자 기능을 회원 role로 흡수하지 않는다(yml 계정 하나로 충분).

**개인정보 인프라(2026-07-17 추가 — 기획 검증에서 발견된 필수 요건):** 회원 도입은 개인정보 수집(이메일·닉네임)이므로 ① **개인정보처리방침 페이지**(`/privacy` — 카카오 앱 등록이 이 URL을 요구하므로 소셜 로그인 구현보다 선행)와 ② **회원 탈퇴**를 함께 도입한다. 탈퇴 시: User·refresh 토큰·개인화 삭제, **댓글은 익명화**(user_id null, 내용 유지 — 사용자 확정), Job 이력은 user_id null(row 잔존), 소셜 unlink 호출. 이슈 055.

## 스키마 (초안 — 구현 시 확정)

```
user            id(PK), provider(GOOGLE|KAKAO), provider_id, email(nullable), nickname, created_at
                UNIQUE(provider, provider_id)
refresh_token   id(PK), user_id(FK), token_hash, expires_at, created_at
job             + user_id(FK, nullable — 익명 Job은 null)
comment         + user_id(FK, nullable — 익명 댓글은 null), 표시용 닉네임은 user 조인
user_favorite   user_id, module_id  (최근 사용·좋아요도 동일 패턴)
```

## 기각한 대안

- **로그인 필수/중심 서비스** — 도구 사이트 이탈 요인, 익명 인프라 폐기 비용, ADR-0009 전면 파기.
- **이메일+비밀번호** — 부속 플로우(인증 메일·재설정) 부담. 소셜 2종으로 국내 커버리지 충분.
- **세션 쿠키** — 크로스 사이트 쿠키가 전제라 취약. 도메인 구매 후 재검토 여지만 남김.
- **단일 장수명 토큰** — 실용은 충분하나 refresh 회전·재발급 설계가 포트폴리오 재료로서 더 가치 있음.

## 결과 (예상)

- 백엔드: `user/` 패키지 신설(entity·repository·service·controller), SecurityConfig에 JWT 필터 추가(기존 permitAll + `/admin/**` Basic 구조 유지), `/api/v1/auth/**` 엔드포인트(콜백·재발급·로그아웃·내 정보).
- Job 생성 시 인증돼 있으면 `user_id` 기록(X-Client-Id 병행 — 익명 쿼터 체계는 그대로).
- 스키마 변경은 전부 Flyway 마이그레이션(ADR-0025).
- Security 변경이므로 완료 검증은 코드 리뷰만으로 부족 — **실브라우저 E2E**(구글·카카오 실로그인, CORS, 토큰 재발급) 필수.
