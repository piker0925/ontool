# ADR-0031 프론트 전용 도구의 사용횟수 집계 방식

## 상태

확정 (2026-07-21)

## 배경

`isFrontendOnly` 도구(순수 클라이언트 계산 — 로또 번호 생성기, UUID, 각종 계산기 등 약 67개, ADR-0020으로 프론트 이전된 것 포함)는 백엔드 실행 API(`/run`, `/upload`)를 아예 거치지 않는다. `ToolStats.use_count`는 이 두 엔드포인트 안에서만 증가하므로, 이 도구들은 아무리 써도 `use_count`가 영원히 0으로 고정되는 버그가 있었다(ADR-0020은 "좋아요·사용수·댓글은 그대로 유효하다"고 서술했으나, 이는 `moduleId` 문자열 조회가 깨지지 않는다는 뜻이었을 뿐 실제 카운트 증가를 검증한 것은 아니었다).

프론트 전용 도구 67개 각각에 "실행 완료" 이벤트를 개별 배선하는 방식은 컴포넌트마다 코드를 심어야 하고, 새 프론트 전용 도구를 추가할 때마다 잊기 쉬운 규칙이 되어 유지비가 계속 든다.

## 결정

**`ToolPage.vue`(도구 상세 페이지 셸) 한 곳에서 중앙집중적으로 처리한다.** 자식 컴포넌트를 감싸는 wrapper에 캡처 단계 이벤트 리스너를 달아 "도구 영역 안에서 실질적 조작이 처음 발생한 시점"에 신설 엔드포인트 `POST /api/v1/tools/{moduleId}/use`(인증 불필요, `use_count` +1)를 1회 호출한다(`front/src/pages/ToolPage.vue` `pingFrontendToolUseOnce`).

- **감지 이벤트**: `click`(단, `<button>` 클릭만 인정 — 텍스트 필드 포커스나 라벨 클릭 제외), `change`(입력 필드는 blur 시점 확정값, select/checkbox는 즉시), `drop`(파일 드롭). **`input`/`keydown`은 제외** — 키 입력마다 발화해 오타 한 글자로도 집계되는 문제가 있었다.
- **집계 시점은 "실행 완료"가 아니라 "실행 의도가 있는 조작"이다**: 버튼을 눌렀다고 결과가 유효했는지까지는 확인하지 않는다. 각 컴포넌트 내부 실행 완료를 정확히 아는 유일한 방법은 컴포넌트마다 이벤트를 심는 것인데, 그 유지비를 이 근사로 피했다.
- **페이지 진입만으로는 집계하지 않는다**: 최초 구현은 마운트 시 즉시 ping을 보냈으나, "들어가기만 해도 오른다"는 지적을 반영해 최초 조작 시점으로 변경했다.
- **이중 집계 방지 예외 목록**: `isFrontendOnly=true`이지만 내부적으로 `useHeavyJob`으로 백엔드 Job API를 직접 호출해 이미 `/upload` 경로로 카운트되는 도구(`pdf-watermark`·`pdf-password`·`pdf-header-footer`·`office-document-convert`·`data-convert`·`code-gen`·`document-generator`)는 `front/src/api/modules.ts`의 `BACKEND_WIRED_FRONTEND_TOOL_IDS`에 등록해 ping 대상에서 제외한다. **새 프론트 전용 도구가 내부적으로 백엔드를 직접 호출하도록 만들 경우, 이 목록에 반드시 추가해야 이중 집계를 피한다.**

## 결과

- `use_count`의 의미가 도구 유형별로 갈린다: Light/Heavy는 "실행 횟수"(클릭마다 +1), 프론트 전용은 "그 방문에서 실제로 조작해봤는가"(방문당 최대 +1). 같은 "사용 N회" 배지가 두 가지를 가리키지만, 정밀 감사 지표가 아니라 인기도 근사치이므로 포트폴리오 성격상 수용했다.
- 검증: `ToolStatsControllerTest`에 비로그인 `/use` 반복 호출 테스트 추가, 로컬 실브라우저로 로또 번호 생성기(클릭 시 +1, 이후 클릭 무반응, 재진입 시 다시 +1)·Cron 표현식(입력 중 미집계, blur 시 +1)·pdf-watermark(방문만으로는 미집계) 각각 확인.
