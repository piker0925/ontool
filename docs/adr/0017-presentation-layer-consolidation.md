# ADR-0017 표현 계층 통합 — 도구 통폐합·프론트 전용 전환은 백엔드 무변경 원칙

## 상태
확정 — 단, '결과'의 `RegexModule`/`DiffModule` 잔존 서술은 이후 **ADR-0020**에서 실제로 제거됨(2026-07).

## 배경

44개 도구는 개수만 많고 밀도가 낮았다. SHA-256은 다중 해시의 순수 부분집합이었고, Base64/URL 인코딩/HTML Entity는 사실상 하나의 "인코더" 화면으로 묶일 수 있었으며, JSON↔YAML/TOML/XML/CSV 4개도 From/To 셀렉터 하나로 대체 가능했다. 또한 정규식 테스터와 텍스트 Diff는 서버 왕복 없이 브라우저에서 즉시 실행하는 편이 UX상 명백히 우월했다(입력 즉시 하이라이트·매치 표시 등).

## 검토한 선택지

| 방식 | 설명 | 문제점 |
|------|------|--------|
| A. 백엔드 모듈까지 물리적으로 병합 | `MultiHashModule`이 `Sha256Module`을 흡수하듯 여러 모듈을 하나로 합침 | 전략 패턴이 훼손되고, `direction`류 분기가 모듈 내부에서 비대해짐. 기존 `ToolStats`/`Comment`/`Job`의 `moduleId` 참조가 전부 무효화되어 히스토리 손실. 매 통합마다 CONTEXT.md·테스트 연쇄 수정 필요 — 이번 라운드 규모(14개 도구군 병렬 작업)에서는 오버엔지니어링 |
| B. 프론트 표현 계층에서만 통합, 백엔드는 무변경 | 통합 페이지가 기존 여러 모듈의 `/run`을 내부적으로 호출 | 백엔드 모듈 수는 안 줄어듦(관리 포인트는 남음) |
| C. 정규식/Diff/TOTP는 프론트 전용으로 완전 전환 | 백엔드 모듈은 호환을 위해 남기되 프론트가 더 이상 호출하지 않음 | 죽은 백엔드 코드가 남음(허용 가능한 트레이드오프로 판단) |

## 결정

**B + C: 프론트 표현 계층에서만 통합, 백엔드 모듈은 유지. 정규식·Diff·TOTP는 프론트 전용으로 전환**

1. **통합 페이지 4종** — 인코더/디코더(`/tools/encoder`), 데이터 포맷 변환(`/tools/data-convert`), 텍스트 유틸(`/tools/text-utils`), 코드 생성기(`/tools/code-gen`). 각 페이지는 형식/모드 셀렉터로 상태를 URL 쿼리와 양방향 동기화하고, 내부적으로 기존 개별 모듈의 `/run` 엔드포인트를 그대로 호출한다.
2. **구 URL 12종**을 `router.redirect`로 유지해 북마크·외부 링크 호환성을 보존.
3. **검색 별칭**: `Module.keywords`를 `{keyword, query?}` 구조로 확장해 "base64", "html entity" 같은 옛 이름으로 검색해도 통합 페이지의 정확한 모드로 딥링크.
4. **정규식 테스터·텍스트 Diff**는 백엔드 호출을 완전히 제거하고 브라우저 로컬 계산(JS `RegExp`, npm `diff`)으로 전환 — `isFrontendOnly: true`로 표시하고 백엔드 `RegexModule`/`DiffModule`은 API 호환을 위해서만 남긴다(신규 호출자 없음).
5. **TOTP**도 프론트 로컬 계산(Web Crypto HMAC-SHA1 + Base32)으로 전환.
6. **유일한 예외**: `Sha256Module`은 물리적으로 삭제하고 `MultiHashModule`이 흡수한다(이전 세션에서 결정·완료) — SHA-256이 다중 해시의 순수 부분집합이라 병합에 따른 손실이 전혀 없는 유일한 케이스였기 때문에 안 A를 이 하나에 한해 적용했다.

## 이유

- 안 A(전면 물리 병합)를 기각한 핵심 이유는 **`ToolStats`/`Comment`가 `moduleId` 문자열을 FK 없이 참조**하기 때문이다(ADR 없음, 스키마 관찰). 모듈을 합치면 기존 사용 횟수·좋아요·댓글이 새 `moduleId`에 연결되지 못하고 고아가 된다. 표현 계층 통합은 이 문제를 피한다 — 백엔드 `moduleId`가 안 바뀌므로 기존 통계·댓글이 그대로 유효하다.
- 정규식/Diff의 프론트 전환은 순수 UX 이유다: 서버 왕복 없는 즉시 하이라이트가 텍스트 처리 도구의 핵심 가치이고, 두 알고리즘 모두 브라우저에서 가벼우며 클라이언트 계산이 문제되지 않는다.
- SHA-256만 물리 병합을 유지한 이유는 "손실 없는 부분집합"이라는 조건이 이 케이스에만 성립하기 때문이다 — 다른 통합 후보(인코더 3종, 변환기 4종)는 각기 다른 파라미터·검증 로직을 가져 물리 병합 시 개별성이 사라진다.

## 결과

- 도구 표시 개수는 44 → 34로 줄었다(코드 생성기 통합으로 추가 2개 흡수, 유틸 카테고리 통합 등 누적).
- 백엔드는 `RegexModule`/`DiffModule`이 죽은 코드로 남아 있다 — 향후 완전 제거 여부는 백로그(`.scratch/backlog-tool-ux.md`)에 기록.
- `ToolStats`/`Comment`의 `moduleId` 무결성은 DB 레벨 FK가 아니라 여전히 애플리케이션 책임이다 — 향후 실제 모듈 삭제/개명 시 반드시 참조 정합성을 수동 확인해야 한다.
